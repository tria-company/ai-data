<objective>
Split the dual role of `projetoId` in worker jobs. Today it both (a) tags the `projeto` DB column and (b) selects which Supabase credential to use via `SUPABASE_PROJECTS`. Introduce a new `db_account` field in the POST /api/scrape request body that carries forward through the BullMQ payloads as `dbAccount` and is used EXCLUSIVELY for credential routing. `projetoId` keeps tagging the `projeto` column unchanged.

Purpose: Enables clients to write into one tenant's DB while tagging rows with a different logical project identifier — decouples "where does this row live" from "what project label does this row carry".

Output: Updated `getSupabaseForJob` signature, updated API entry point, updated worker job interfaces and enqueue paths, with strict backward compatibility for legacy callers and in-flight jobs.
</objective>

```typescript
export function getSupabaseForJob(
  dbAccount: string | null | undefined,
  projetoIdFallback?: string | null,
```

3. Else return the legacy `getSupabase()` client (unchanged last-resort fallback).

Update the JSDoc comment above the function to reflect the new semantics. It should explain:
- `dbAccount` is the PRIMARY credential router key (looked up in `SUPABASE_PROJECTS`)
- `projetoIdFallback` exists for backward compat with legacy jobs/callers that were using projetoId as the DB selector
- Neither argument is ever written to a DB column — this function only picks which client to return
- Both args null/empty → legacy `getSupabase()` client

DO NOT:
- Rename or remove `getSupabaseForProject` or `getSupabaseAnonKeyForProject` (still used by their current callers).
