<objective>
Migrate `scrapeProfile` (profile-worker.ts) and `scrapePostDetails` (post-worker.ts) so that every per-tenant DB write lands in the correct Supabase project. We achieve this by introducing a single dispatcher helper `getSupabaseForJob(projetoId)` in `lib/supabase.ts` and threading a local `db` client into both scrape functions.

Purpose: Multi-tenant correctness. Today the workers import the legacy singleton `supabase` from `lib/supabase.ts`, which always points at the default project. After quick-task `260410-mor` introduced `getSupabaseForProject`, per-job routing is possible but not yet used. Without this migration, jobs enqueued with a `projetoId` write to the wrong Supabase instance.

Output: `getSupabaseForJob` helper + two refactored worker files where all DB writes inside the scrape functions go through `db` (tenant-aware) instead of `supabase` (global singleton). Fully backward compatible — jobs with `projetoId == null` still hit the legacy client.
</objective>

Insert exactly this block at the end of the file (preserving the trailing newline):

```typescript
/**
 * Returns the appropriate Supabase client for a job based on projetoId.
 *
 * - If projetoId is a non-empty string: routes to the tenant-specific client
 *   via getSupabaseForProject().
 * - If projetoId is null/undefined/empty string: falls back to the legacy
 *   getSupabase() client (backward compat for jobs enqueued before
 *   multi-tenant support).
 *
 * Use this inside worker job processors where projetoId comes from job.data
 * and may be null for legacy jobs still in the queue.
 */
export function getSupabaseForJob(projetoId: string | null | undefined): SupabaseClient {
  if (projetoId && projetoId.trim() !== '') {
    return getSupabaseForProject(projetoId);
  }
  return getSupabase();
}
```
